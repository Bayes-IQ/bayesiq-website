Every data team we've worked with has the same assumption: the numbers in the dashboard are correct. They're not.

## The silent failure pattern

Data quality doesn't fail like a server outage. There's no alert, no error page, no angry Slack message. Instead, it degrades slowly:

- A logging spec changes but the implementation doesn't get updated
- A pipeline starts silently dropping rows after a schema migration
- A metric definition drifts from its original meaning as the product evolves

By the time someone notices, weeks or months of decisions have been made on bad data.

## A real example

In a recent engagement, we found **3 event types** that had stopped firing on mobile web. The root cause? A race condition in the analytics SDK initialization introduced 4 months earlier. No one noticed because the web numbers were still "in the right ballpark."

The impact: revenue metrics were underreporting by **12%**. Finance and analytics had been disagreeing for months, and both teams assumed the other was wrong.

## What a data audit actually finds

| Finding type | How often we see it | Typical impact |
|---|---|---|
| Missing events | ~80% of audits | Metric gaps, underreporting |
| Null/malformed fields | ~70% of audits | Broken downstream queries |
| Duplicate events | ~50% of audits | Inflated metrics |
| Stale metric definitions | ~60% of audits | Misleading dashboards |

These aren't edge cases. They're the norm.

## What to do about it

Start with a structured audit. Not a vague "data quality review" — a systematic check of your telemetry against your logging spec, your pipeline transformations against your schema, and your metric definitions against the underlying data.

If you want to try this yourself, here's a starting checklist:

- [ ] Compare your logging spec to actual fired events (field by field)
- [ ] Check for null rates on required fields
- [ ] Validate that pipeline output row counts match input expectations
- [ ] Confirm metric definitions match their SQL implementations
- [ ] Look for duplicate events (check idempotency)
