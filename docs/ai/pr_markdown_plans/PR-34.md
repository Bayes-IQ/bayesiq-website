## Goal

Add a business event preview panel that shows how metric changes, restatements, and other business events flow through BayesIQ's governance system. Each vertical with business events gets a preview section between feedback threads and CTA. The component visually differentiates between preview (pending/deferred) and resolved (approved/rejected) states, demonstrating that every metric change is reviewed before it affects outputs.

## Design Decisions

### D-001: Vertical-scoped filtering via event_id prefix

Business events are cross-vertical in the governance payload. To show per-vertical events, filter `businessEventById` by checking if `event_id` starts with the vertical slug (e.g., `fintech_*` for fintech-gf). The mapping from vertical slug to event_id prefix is:

```ts
function verticalPrefixFromSlug(slug: string): string {
  // "fintech-gf" -> "fintech", "hospital" -> "hospital"
  return slug.replace(/-gf$/, "");
}
```

This is computed in the server component (page.tsx) and passed as a plain array to the client component.

### D-002: Preview vs. approved visual distinction

Following the TrustBadge pattern from GF-18:
- **Pending/Deferred** = "preview" state: amber/gray badge, dashed border on card
- **Approved** = solid green badge, solid border on card
- **Rejected** = red badge, red-tinted border on card

The card border style communicates the governance state at a glance without reading text.

### D-003: Expand/collapse pattern from FeedbackThread

Reuse the same expand/collapse pattern as `FeedbackThread.tsx`:
- Collapsed: event_id (humanized), approval status badge, reviewer name, relative time
- Expanded: review note, timestamps (requested/resolved), record origin, source_approval_id

### D-004: Insertion point in page.tsx

Insert between feedback threads section and GoldenFlowsCTA. The section only renders if there are events for the current vertical.

### D-005: TrustBadge reuse for status display

Each card's status badge reuses the `TrustBadge` component with `size="sm"` and no `onClick` (no governance detail drill-down for business events in this PR — that can be added later when the governance detail panel supports the `"event"` objectType).

## Implementation

### 1. BusinessEventPreview.tsx

```tsx
"use client";

import { useState } from "react";
import type { BusinessEventItem } from "@/lib/governance";
import TrustBadge from "./TrustBadge";

interface BusinessEventPreviewProps {
  item: BusinessEventItem;
}
```

**Collapsed view:**
- Humanized event_id: replace underscores with spaces, title-case
- TrustBadge with approval_status (size="sm", no onClick)
- Reviewer display_name
- Relative time from ts_requested

**Expanded view:**
- Review note (if present, in `bg-bayesiq-50 rounded-lg p-3` block)
- Timestamps: requested and resolved (formatted dates)
- Record origin badge
- Source approval ID as `<code>` text

**Card border styling:**
```ts
function borderStyle(status: BusinessEventItem["approval_status"]): string {
  switch (status) {
    case "pending":
    case "deferred":
      return "border-dashed border-amber-300";
    case "approved":
      return "border-solid border-green-300";
    case "rejected":
      return "border-solid border-red-300";
  }
}
```

### 2. BusinessEventList.tsx

```tsx
"use client";

import type { BusinessEventItem } from "@/lib/governance";
import BusinessEventPreview from "./BusinessEventPreview";

interface BusinessEventListProps {
  events: BusinessEventItem[];
}
```

**Grouping:** Events grouped by status category:
1. "Pending Review" (pending + deferred)
2. "Resolved" (approved + rejected)

Within each group, sorted by `ts_requested` descending (most recent first).

**Empty state:** If events array is empty, component returns null (section not rendered).

### 3. Page.tsx Wiring

In the server component, after the existing feedback items extraction:

```tsx
// Business events (GF-20) — filter by vertical prefix
const verticalPrefix = slug.replace(/-gf$/, "");
const businessEvents = governance?.businessEventById
  ? Array.from(governance.businessEventById.values()).filter(
      (e) => e.event_id.startsWith(verticalPrefix + "_")
    )
  : [];
```

In the JSX, insert between feedback threads and CTA:

```tsx
{/* Business Events — GF-20 */}
{businessEvents.length > 0 && (
  <section className="mt-12">
    <h2 className="text-xl font-bold tracking-tight text-bayesiq-900 mb-4">
      Business Events
    </h2>
    <p className="text-sm text-bayesiq-500 mb-4">
      Metric changes and restatements flowing through governance review.
    </p>
    <BusinessEventList events={businessEvents} />
  </section>
)}
```

### 4. Serialization

`BusinessEventItem` objects contain only primitives (strings, nulls). No Maps or special types. The array can be passed directly from server to client component as a prop — no serialization issues.

  - **Mitigation:** Prefix extraction is isolated in one function; easy to replace with explicit vertical_id field if upstream adds one
- **Risk:** Business events payload empty for some verticals
  - **Mitigation:** Section conditionally rendered only when events exist; empty state handled gracefully
